The Problem We're Solving
Kytchens generates rich operational data across 7 sources but only one source — UrbanPiper orders — lives in a database. Everything else (inventory, complaints, photo evidence, financials, staff, offline sales) sits in Google Sheets, email inboxes, and third-party portals. Nothing is connected. There is no single place to answer the question: "how did this kitchen perform today, and why?" The architecture below changes that.
What exists and works ✓
  • UrbanPiper orders flow into ClickHouse via live webhook — 1M+ order rows
  • aggregated_data table already pre-aggregates GMV, orders by brand × location × date with correct types
  • Item-level sales (transformed_itemised_sales) has category, Item_ID, Item_Ref_ID
  • Order state transitions (prime_order_statuses) exist — KPT is calculable
What's broken ✗
  • All financial columns in ClickHouse are stored as String — every revenue calculation requires string parsing
  • Order status timestamps are String — KPT computed from string subtraction, a silent error source
  • All joins on brand/location text strings — case drift produces wrong aggregations silently
  • No brand_id or kitchen_id UUID anywhere in the schema
What needs to be built ○
  • order_id_map — the bridge between UP Order ID (Bravery) and Aggregator Order ID (Complaints)
  • Inventory pipeline — hourly SupplyNote email → timestamped database table, not a Google Sheet
  • Evidence pipeline — Bravery CSV (photo + maker + checker) + Complaints email → one row per order
  • dim_brand_kitchen — brand map sheet with typed NUMERIC fields replacing text

🔑 The most important table: order_id_map
Bravery uses UP Order ID as its key. Restaverse complaints use Aggregator Order ID (Swiggy/Zomato's own ID). These are different numbers for the same order. The order_id_map table holds both together — it's the only place they coexist. Every join between evidence and complaints runs through this table. Build it first.
🗄️ ClickHouse stays, Supabase is added
ClickHouse handles 177M+ order rows at speed — it stays as the orders analytics engine. Supabase (Postgres) becomes the operational layer for all new modules: inventory, evidence, complaints, financials, people. A Semantic layer (Layer 4) unifies both into one query interface for dashboards and the AI agent.
📸 Evidence expires in 14 days
Bravery photos are deleted from their server 14 days after capture. photo_expires_at is stored as a computed column in every evidence row. An alert fires 48 hours before expiry. Without this, complaint evidence is silently lost — and there's no way to know a photo ever existed.
💰 Financial types: the biggest quick win
The ClickHouse migration stored all financial columns as String — Sub_Total_Amount, Merchant_Discount, Total_Taxes etc. Every revenue query today parses strings before doing arithmetic. Correcting these to NUMERIC(12,2) at the Staging layer is the fastest improvement to data reliability.

The Universal ID Map
Six identifiers, three systems, one consistent naming convention.
up_order_id
UrbanPiper's internal order ID. Bravery CSV calls it order_id. ClickHouse calls it Order_ID.
fact_orders · fact_items · fact_order_evidence · order_id_map
aggregator_order_id
Swiggy/Zomato's own ID. Restaverse complaints call it order_id. Bravery calls it aggregator_order_no. ≠ up_order_id.
order_id_map · fact_order_evidence · raw_complaints
up_store_ref_id
The brand×kitchen resolver. Called quint_id in Brand Map, Store_ID in ClickHouse, res_id in Restaverse complaints. Always TEXT, never INT.
dim_brand_kitchen · order_id_map · raw_complaints.res_id
sku_id
Our internal UUID for each SKU. Resolves from three source IDs: item_id (UrbanPiper), item_ref_id (aggregator + stock events), sn_sku_code (SupplyNote).
dim_skus · fact_items · fact_stock_snapshot · fact_stock_events
keka_employee_id
Resolves free-text names in Bravery CSV: maker ("ganesh g"), Checker ("Pawan B"), order_captured_by ("Chembur Supervisor").
dim_employees · fact_order_evidence (×3 cols) · fact_pos_sales
The end state in one sentence: Any person in Kytchens — a KM asking why their complaint rate spiked, a brand partner checking their GMV, the Founder's Office calculating monthly settlements — types a question in plain English and gets a correct, cited answer from a single system backed by clean, connected, typed data. That's what this architecture is for.
