Data Sources
Seven Sources — One System
What each source holds, how it arrives, and what needs to change. Only UrbanPiper data arrives via webhook. Everything else arrives via email, CSV, or manual entry.
Webhook ✓ 🍽️
Orders · POS · Aggregator
UrbanPiper → ClickHouse
Only source with live webhook integration. Swiggy & Zomato orders aggregated via UrbanPiper, buffered through BraveryKOA server, ingested into MySQL on AWS, ETL'd into ClickHouse. Five live tables. aggregated_data already functions as a Layer 3 KPI table with correct Decimal types. All financial columns were degraded to String during ClickHouse migration — needs type correction at staging.
Hourly Email 📦
Inventory ERP
SupplyNote
Point-in-time stock snapshot delivered via scheduled hourly email as CSV attachment. Manual download → clean → paste into brand-specific Google Sheet per location. OOS checked by manually comparing current stock vs par level in a separate sheet. Stock in/out transaction data is available via UrbanPiper webhook (configurable) — this is more valuable than snapshots as it gives movement history.
No DB. Every snapshot overwrites the last. OOS history is zero. Manual sheet process is error-prone.
Alt-day CSV 📸
Photo Evidence · Maker · Checker
Bravery Photo App
Every packed order is photographed before dispatch. Bulk CSV downloaded every alternate day. File contains: order_id (= UP Order ID), aggregator_order_no (= Aggregator Order ID), maker, Checker, order_captured_by (supervisor), kitchen_name, brand_name, photo filename, capture timestamp, is_order_image_captured flag. Photos stored on Bravery server for 14 days then deleted.
Photo metadata not in DB. Evidence window closes silently. Alt-day import means up to 48-hour lag.
Email CSV 💬
Complaints · ORM
Restaverse
Complaints received as a daily email with CSV attachment — not via API. File contains: order_id (= Aggregator Order ID), res_id (= quint_id = Store ID), brand, subzone, platform, reason (raw), comment, media URL, complaint_item_names, cart_item_names, requested_refund_amount, complain_status (DISMISSED / UNRESOLVED), order_created_at, complaint_created_at. Currently processed manually — cleaned output is stale.
Manual process. Cleaned output stale. No automated ingestion. Complaint resolution history not tracked.
Sheet 💰
Commercial Config
Brand Map Sheet
Google Sheet: brand name, location, quint_id (= up_store_ref_id = Store ID), MG rental, revenue share %, go-live date. The existing brand_revenue table in ClickHouse has only brand + revenue_model — no commercial config. quint_id is the key that bridges every brand/location to UrbanPiper store references, SupplyNote outlet names, and the complaints res_id.
MG and rev share stored as text with ₹ and % symbols. No types, no FK, no version control.
HRMS 👥
People Data
Keka
All employee records — name, designation, kitchen assignment, reporting structure. Maker and checker names in the Bravery CSV (e.g. "ganesh g", "Deepak R", "Pawan B") are free-text strings. order_captured_by (supervisor) is also free text. These need to be resolved to Keka employee IDs for staff performance attribution to work. One-way sync from Keka into dim_employees.
Zero integration. Staff performance analytics impossible without Keka ID resolution.
Finance 📒
Invoicing
Tally
All invoicing and financial accounting. Settlement figures calculated from Brand Map sheet cannot be auto-pushed to Tally. The monthly_settlement table in the Analytics layer is designed to produce the exact format Tally needs — bridging UrbanPiper GMV to Tally invoices for the first time, eliminating the current manual reconciliation.
Manual reconciliation between UrbanPiper GMV and Tally invoices every month.
Manual Entry ✍️
Offline Revenue
POS + Manual Sales
POS sales, rental brand sales, event sales — entered manually in varied formats across different sheets. A prior POS schema existed with bill-level detail including cost price, GST breakdown (5/12/18%), net sales, employee attribution, and payment status. These are real revenue events that currently sit entirely outside the GMV picture and cannot be aggregated with UrbanPiper sales.
No canonical schema enforced. Cannot roll into total revenue. COGS data exists but unused.

Rules + KPIs
Golden Rules & KPI Reference
Rules prevent problems we already have. KPIs define exactly what we measure and from where.
1
Raw is immutable. Layer 0 is append-only forever. Every source file stored exactly as received with ingested_at timestamp. Fix type problems at staging, never in raw.
2
Financial columns are always NUMERIC. ₹ signs and % stripped at the staging boundary. The ClickHouse type regression (String financials) is corrected here.
3
Timestamps always TIMESTAMPTZ in IST. All date strings parsed at staging. KPT calculated from typed timestamps, never from strings.
4
up_order_id ≠ aggregator_order_id. Bravery order_id = UP Order ID. Complaints order_id = Aggregator Order ID. Mixing them is the #1 zero-row join cause.
5
res_id = quint_id = up_store_ref_id. One concept, one column name in the schema: up_store_ref_id. Always stored as TEXT, never as INT.
6
Brand and kitchen are UUID FKs in Mart and above. Text string joins on brand/location names produce wrong aggregations when names drift. UUIDs prevent this.
7
KPT is always computed: mfr_time − ack_time in minutes. Never stored as a pre-computed value from an upstream source. Calculation only from typed TIMESTAMPTZ columns.
8
Every pipeline is idempotent. Running an import twice produces zero change. UPSERT on natural keys always. Prevents double-row bugs on re-imports.
9
Photo expiry is tracked. photo_expires_at = photo_captured_at + 14 days. An alert fires before this date. Evidence is never silently lost from Bravery's server.
10
Inventory snapshots are appended, not overwritten. Every hourly email pull adds rows to fact_stock_snapshot with its timestamp. OOS history requires full history.
11
Maker/checker names resolved at staging. "ganesh g" → keka_employee_id. Raw name kept as fallback. Unresolved rows go to stg_unresolved — never silently attributed to nobody.
12
Semantic layer is read-only for the AI agent. All queries go through kytchens_semantic views only. Column names are full English phrases — they are the documentation.
